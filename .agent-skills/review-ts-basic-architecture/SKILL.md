# 基本アーキテクチャ・コード品質レビュー
## 概要

このスキルは、プルリクエストやgit diffの基本的なアーキテクチャとコード品質に特化したコードレビューを提供する。主に以下の3つの領域に焦点を当てている：

### 可読性

- 関数名の命名規則（動詞＋名詞）
- if文のネストと早期リターン
- 変数の命名規則（アッパースネークケース、ローワーキャメルケース）
- プライベート関数の定義

### 拡張性・保守性

- ロジックの分離と統合の適切性
- 型定義（`any`型の禁止、`type`エイリアスの使用）
- 未使用コードの削除
- React.FCの使用禁止
- 型アノテーションの明示

### 信頼性

- エラーハンドリング（try-catch、適切なエラーメッセージ、エラーログ、リトライ処理）
- バリデーション（入力バリデーション、型安全性、境界値チェック、Null/Undefinedチェック）
- データ整合性（トランザクション、競合状態、データの一貫性）
- ロジックの正確性（エッジケース、無限ループ、早期リターン、未使用コード、重複コード）
- 非同期処理の実装
- テストコードの実装

レビューでは、重要度（Critical/High/Medium/Low）別に分類された詳細なレポートを生成し、具体的な修正提案とベストプラクティスを含める。

**注意**: セキュリティ（機密情報の管理、インジェクション攻撃など）とパフォーマンス（DB操作の最適化、非同期処理の効率化など）は、`security-performance-review`スキルでカバーされるため、このスキルでは扱わない。

## レビュープロセス

徹底的なコードレビューを実施するために、以下の手順に従う：

### 1. レビュー範囲の特定

レビューが必要な対象を決定する：

- **PRの場合**: `gh pr diff <pr-number>` で完全なdiffを取得
- **未コミットの変更の場合**: `git diff` または `git diff --staged` を使用
- **コミット済みの変更の場合**: `git diff <base-branch>...HEAD` を使用

ユーザーが指定していない場合は、どの範囲をレビューしたいか尋ねる。

### 2. プロジェクトコンテキストの理解

レビューを開始する前に、プロジェクトの規約と制約を理解する：

1. **CLAUDE.mdを読む**: このファイルにはプロジェクト固有のコーディング標準、アーキテクチャパターン、規約が含まれている。最初に読み込んで、何をチェックすべきか理解する。

2. **コードベース構造を理解する**: CLAUDE.mdで指定されているディレクトリ構造の規約と依存関係のルールを確認する。

3. **重要領域に注目する**: CLAUDE.mdで高優先度として言及されている領域に特に注意を払う。

### 3. 体系的なレビューの実施

`references/checklist.md`を包括的なレビューチェックリストとして使用する。以下の3つの領域を体系的にチェックする：

#### 優先度1: 可読性

- 関数名の命名規則
- if文のネストと早期リターン
- 変数の命名規則
- プライベート関数の定義

#### 優先度2: 拡張性・保守性

- ロジックの分離と統合
- 型定義
- 未使用コードの削除
- React.FCの使用禁止
- 型アノテーションの明示

#### 優先度3: 信頼性

- エラーハンドリング
- バリデーション
- データ整合性
- ロジックの正確性
- 非同期処理
- テストコード

コードの変更に基づいて、関連するセクションを読み込んで体系的に評価する。

**重要**: レビュー中は常にこのファイルを読み込んで参照すること。内容を記憶に頼らない。

### 4. 変更ファイルごとの分析

diffの各ファイルについて：

1. **目的を理解する**: この変更は何を実現しようとしているのか？
2. **CLAUDE.mdのルールと照合する**: プロジェクトの規約に従っているか？
3. **チェックリスト項目を適用する**: 関連するチェックリストのセクションを確認
4. **具体的な問題を記録する**: ファイルパス、行番号、問題の説明、重要度を記録

### 5. レビューレポートの生成

以下の「レポート形式」セクションで説明されている形式で包括的なレポートを作成する。

### 6. コード修正後の検証

指摘した問題をユーザーが修正した後、以下のコマンドを実行して、修正が正しく機能しているか検証する：

```bash
yarn build
yarn test --run
yarn lint:llm [対象ファイル]
```

- **`yarn build`**: ビルドエラーがないことを確認
- **`yarn test --run`**: すべてのテストがパスすることを確認
- **`yarn lint:llm [対象ファイル]`**: LLMベースのlintチェックを実行

エラーが発生した場合は、エラー内容を分析して追加の修正提案を行う。

## レポート形式

以下の構造でレビューレポートを生成する：

### サマリー

- **検出された問題の総数**: <数値>
- **Critical**: <件数>
- **High**: <件数>
- **Medium**: <件数>
- **Low**: <件数>
- **全体的な評価**: コード品質の簡潔な2〜3文のまとめ

### 重要度別の問題

各重要度レベル（Critical → High → Medium → Low）について、以下の形式で問題を列挙する：

#### [重要度] <問題カテゴリ>

**ファイル**: `<file_path>:<line_number>`

**問題**: 問題の明確な説明

**影響**: 潜在的な結果やリスクの説明

**現在のコード**:

```typescript
// 問題のあるコードスニペットを表示
```

**修正提案**:

```typescript
// 説明付きで修正されたコードを表示
```

**ベストプラクティス**: 追加のコンテキストや推奨されるパターン（該当する場合）

---

### 問題が見つからなかった場合

重要度カテゴリで問題が見つからなかった場合は、「✅ [重要度]の問題は見つかりませんでした」と記載する。

### 追加の所見

ポジティブな観察結果を含める：

- よく実装されているパターン
- 従われている良い慣行
- 称賛に値する改善

## リソース

### references/checklist.md

基本的なアーキテクチャとコード品質に関する包括的なチェックリスト。このファイルには以下が含まれている：

- **可読性**: 関数名、if文のネスト、変数の命名規則、プライベート関数
- **拡張性・保守性**: ロジックの分離、型定義、未使用コード、React.FC、型アノテーション
- **信頼性**: エラーハンドリング、バリデーション、データ整合性、ロジックの正確性、非同期処理、テストコード

各チェック項目は重要度（Critical/High/Medium/Low）で分類されており、レビューの優先度付けに役立つ。

**使用方法**: すべてのレビューの開始時にこのファイルを読み込み、コードの変更に基づいて関連するセクションを体系的に確認する。

## 効果的なレビューのためのヒント

1. **規約遵守を最優先する**: CLAUDE.mdで定義されているプロジェクト固有の規約に違反していないか確認する
2. **可読性を重視する**: コードは書く時間より読む時間の方が長いため、可読性の問題は早期に修正する
3. **型安全性を確保する**: `any`型の使用や型アノテーションの欠落は、将来のバグの温床になる
4. **具体例を提供する**: 問題を指摘するだけでなく、適切な修正方法も示す
5. **コンテキストを考慮する**: プロジェクトの制約を考えると、いくつかの「違反」は許容できる場合がある
6. **CLAUDE.mdを参照する**: 変更を提案する際は、CLAUDE.mdの特定のルール（「可読性の規約」「拡張性・保守性の規約」「信頼性の規約」）を引用して推奨事項を正当化する
7. **セキュリティ・パフォーマンスは別スキル**: セキュリティやパフォーマンスに関する問題を見つけた場合は、`security-performance-review`スキルの使用を推奨する
