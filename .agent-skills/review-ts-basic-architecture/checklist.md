# 基本アーキテクチャ・コード品質チェックリスト

このファイルには、PRレビュー時に確認すべき基本的なアーキテクチャとコード品質に関する具体的なチェック項目を記載する。

**注意**: セキュリティ（機密情報の管理、インジェクション攻撃など）とパフォーマンス（DB操作の最適化、非同期処理の効率化など）は、`security-performance-review`スキルでカバーされるため、このチェックリストでは扱わない。

## 1. 可読性

### High

- [ ] **関数名の命名規則**: 関数名は責務を適切に表し、動詞＋名詞の形式で命名されているか（例：`getUserById()`）
- [ ] **if文のネスト**: if文が複数ネストする場合、早期リターンを用いて処理を簡潔にしているか
- [ ] **変数の命名規則**: グローバル変数はアッパースネークケース、それ以外の変数はローワーキャメルケースで命名されているか（例：`MAX_RETRY_COUNT`、`userName`）

### Medium

- [ ] **プライベート関数**: ほかのファイルから呼び出していない関数をプライベート関数として定義しているか

## 2. 拡張性・保守性

### Critical

- [ ] **any型の禁止**: 厳密な型定義を使用し、`any`型を使用していないか、すべての変数に適切な型定義があるか

### High

- [ ] **ロジックの分離と重複コード**: 複数回呼び出されるロジックが適切に関数や定数に分離されているか。一度しか使用されないロジックは、呼び出し元の処理に統合されているか
- [ ] **型定義の統一**: 型定義の方法は統一されており、すべて`type`エイリアスで定義されているか
- [ ] **型アノテーションの明示**: 関数の引数に型アノテーションが明示されているか

### Medium

- [ ] **未使用コード**: 定義後に一度も使用されていない変数、定数、関数が存在しないか
- [ ] **React.FCの使用禁止**: コンポーネント実装時にReact.FCの使用は禁止されており、JSX.Elementを使用しているか

## 3. 信頼性

### 3.1 エラーハンドリング

#### Critical

- [ ] **未処理の例外**: `try-catch`が必要な非同期処理や外部API呼び出し、バックエンドで例外が発生しうる処理でエラーハンドリングが欠落していないか

#### High

- [ ] **適切なエラーメッセージ**: Reactコンポーネントでユーザー向けのエラーメッセージが表示されているか
- [ ] **エラーログ**: `./utils`ディレクトリの`logger.ts`を利用して`logger.error`で適切にログ出力されているか
- [ ] **リトライ処理**: 通信処理にリトライ機構が実装されているか

### 3.2 バリデーション

#### Critical

- [ ] **入力バリデーション**: リクエストで受信したパラメーターに適切なバリデーションが実装されているか

#### High

- [ ] **境界値チェック**: 配列のインデックスアクセス、数値の範囲チェックが適切に行われているか
- [ ] **Null/Undefinedチェック**: オプショナルな値へのアクセス前に適切にチェックされているか

### 3.3 データ整合性

#### High

- [ ] **トランザクション**: 複数のDB操作が論理的に一つの単位である場合、トランザクションを使用しているか
- [ ] **競合状態**: 並行処理で共有リソースへのアクセスが適切に制御されているか
- [ ] **データの一貫性**: 関連するデータの更新・削除が適切に行われているか（例: 外部キー制約、カスケード削除）

### 3.4 ロジックの正確性

#### High

- [ ] **エッジケース**: 空配列、空文字列、`null`などのエッジケースが考慮されているか
- [ ] **無限ループ**: ループ条件が適切に設定されており、無限ループの可能性がないか

### 3.5 テスト

#### High

- [ ] **テストコード**: テストコードが存在しない場合、正常系・異常系を網羅するテストが実装されているか。E2Eテストコードは`./cypress/e2e`ディレクトリに配置されているか

## レビュー時の注意事項

### 重要度の判断基準

- **Critical**: データ損失、システムクラッシュの可能性がある問題
- **High**: バグ、明確な規約違反
- **Medium**: 可読性・保守性の問題
- **Low**: コードスタイルの改善提案

### 具体的な修正提案

問題を指摘するだけでなく、具体的な修正コード例を提供する

### ベストプラクティス

より良い実装方法やパターンが存在する場合は提案する

### 影響範囲

問題が及ぼす影響やリスクを明確に説明する

### プロジェクト規約との照合

CLAUDE.mdで定義されている規約（可読性、拡張性・保守性、信頼性）に違反していないか確認する

**注意**: CLAUDE.mdの「安全性の規約」は`security-performance-review`スキルでチェックされる。
