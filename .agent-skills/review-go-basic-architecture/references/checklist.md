# Go アーキテクチャ・コード品質レビュー チェックリスト

このチェックリストは、Goプロジェクトのコードレビューを体系的に実施するためのガイドラインである。各項目は重要度（Critical/High/Medium/Low）別に分類されている。

## 優先度1: 可読性 (Readability)

コードが直感的で、Goの慣習（イディオム）に従っているかを確認する。

### 命名規則
- **[High] パッケージ名**: 小文字の単一単語であるか？（snake_case や camelCase を避ける）
- **[High] エクスポート**: 外部から参照されるべきものは大文字で始まっているか？（内部用は小文字）
- **[Medium] レシーバ名**: 短く（通常1〜3文字）、一貫性があるか？（`self` や `this` を避ける）
- **[Medium] 変数名**: スコープが狭い場合は短く、広い場合は説明的な名前か？
- **[Low] インターフェース名**: メソッドが1つの場合、`-er` サフィックス（例: `Reader`, `Writer`）が付いているか？

### 制御構造
- **[High] 早期リターン**: `if err != nil` の後に `else` を使わず、早めにリターンしているか？
- **[Medium] switch文**: 複雑な `if-else` チェーンの代わりに `switch` が検討されているか？

### コメントとドキュメント
- **[Medium] パッケージコメント**: パッケージの冒頭に `// Package ...` 形式のコメントがあるか？
- **[Medium] エクスポートされたシンボル**: すべてのエクスポートされた関数、型、変数にコメントがあるか？
- **[Low] TODOコメント**: 適切な説明と担当者が記載されているか？

---

## 優先度2: 拡張性・保守性 (Maintainability)

長期的なメンテナンスと変更の容易さを確認する。

### アーキテクチャとパッケージ構成
- **[Critical] 循環参照**: パッケージ間で循環参照が発生していないか？
- **[High] internal パッケージ**: 外部に公開すべきでないコードが `internal/` に適切に配置されているか？
- **[Medium] 抽象化の度合い**: 必要以上にインターフェースを多用していないか？（「インターフェースを要求し、構造体を返す」原則）

### 依存関係
- **[High] Context の伝搬**: `context.Context` が関数の第1引数として渡されているか？（構造体の中に保持していないか）
- **[Medium] 依存性の注入**: ハードコードされた依存関係の代わりに、コンストラクタ等で注入されているか？

### 初期化
- **[Medium] init 関数**: `init` 関数の使用は最小限に抑えられているか？（予期せぬ副作用を避ける）

---

## 優先度3: 信頼性 (Reliability)

バグ、レースコンディション、リソースリークを防ぐ。

### エラーハンドリング
- **[Critical] エラーの無視**: `_ = f()` のようにエラーを無視していないか？
- **[High] エラーラッピング**: エラーを返す際にコンテキストを付加しているか？（`fmt.Errorf("...: %w", err)`）
- **[Medium] センチネルエラー vs 型アサーション**: 適切なエラー判定方法が使われているか？

### リソース管理
- **[High] defer の使用**: ファイル、ネットワーク接続、ロックなどの解放に `defer` が使われているか？
- **[High] defer のタイミング**: ループ内で `defer` を使ってリソースを使い果たしていないか？

### 並行処理 (Concurrency)
- **[Critical] データレース**: 複数のゴルーチンから同期なしで共有変数にアクセスしていないか？
- **[High] ゴルーチンのリーク**: ゴルーチンが確実に終了する仕組み（context やチャネルのクローズ）があるか？
- **[Medium] チャネルの使用**: チャネルのバッファサイズが適切か？（デッドロックの可能性はないか）

### データ整合性と安全性
- **[High] スライスの境界**: インデックスアクセス前に長さのチェックがあるか？
- **[Medium] nil チェック**: ポインタをデリファレンスする前に `nil` チェックがあるか？
- **[Medium] マップの初期化**: `nil` マップへの書き込みを行っていないか？

### テスト
- **[High] テストの有無**: 新しいロジックに対してテストが書かれているか？
- **[Medium] テーブル駆動テスト**: 複数のケースを効率的にテストする形式になっているか？
- **[Low] ヘルパー関数**: テストコードの重複を避けるために `t.Helper()` が使われているか？
