# Go 基本アーキテクチャ・コード品質レビュー
## 概要

このスキルは、プルリクエストやgit diffの基本的なアーキテクチャとコード品質に特化したコードレビューを提供する。主に以下の3つの領域に焦点を当てている：

### 可読性

- 命名規則（CamelCase、短く簡潔なレシーバ名、パッケージ名の小文字単一単語）
- 早期リターン（`if err != nil { return err }` の後で `else` を使わない）
- エクスポート（大文字）と非エクスポート（小文字）の適切な使い分け
- ドキュメントコメント（`// Package ...`, `// Function ...`）

### 拡張性・保守性

- インターフェースの適切な使用（必要以上に抽象化しすぎない）
- パッケージ構成の適切性
- `context.Context` の適切な伝搬（最初の引数として渡す）
- `init` 関数の使用を控える
- 依存性の注入（DI）の活用

### 信頼性

- エラーハンドリング（エラーを無視しない、適切なエラーラッピング `fmt.Errorf("...: %w", err)`）
- リソースの解放（`defer` を使用した `Close()` などの呼び出し）
- 並行処理の安全性（レースコンディション、デッドロック、ゴルーチンのリーク防止）
- スライスやマップの初期化と境界値チェック
- テストコードの実装（テーブル駆動テストの活用）

レビューでは、重要度（Critical/High/Medium/Low）別に分類された詳細なレポートを生成し、具体的な修正提案とベストプラクティスを含める。

**注意**: セキュリティ（機密情報の管理、インジェクション攻撃など）とパフォーマンス（プロファイリング、アロケーションの最適化など）は、`security-performance-review`スキルでカバーされるため、このスキルでは扱わない。

## レビュープロセス

徹底的なコードレビューを実施するために、以下の手順に従う：

### 1. レビュー範囲の特定

レビューが必要な対象を決定する：

- **PRの場合**: `gh pr diff <pr-number>` で完全なdiffを取得
- **未コミットの変更の場合**: `git diff` または `git diff --staged` を使用
- **コミット済みの変更の場合**: `git diff <base-branch>...HEAD` を使用

ユーザーが指定していない場合は、どの範囲をレビューしたいか尋ねる。

### 2. プロジェクトコンテキストの理解

レビューを開始する前に、プロジェクトの規約と制約を理解する：

1. **GEMINI.md または CLAUDE.md を読む**: このファイルにはプロジェクト固有のコーディング標準、アーキテクチャパターン、規約が含まれている。最初に読み込んで、何をチェックすべきか理解する。

2. **コードベース構造を理解する**: `go.mod` を確認し、主要なディレクトリ構造（`cmd/`, `internal/`, `pkg/` など）を確認する。

### 3. 体系的なレビューの実施

`references/checklist.md`を包括的なレビューチェックリストとして使用する。以下の3つの領域を体系的にチェックする：

#### 優先度1: 可読性

- 命名規則（変数、関数、型、パッケージ、レシーバ）
- 早期リターン
- コメントとドキュメント
- ファイル構成

#### 優先度2: 拡張性・保守性

- インターフェース設計
- パッケージ間の依存関係
- Context の使用
- 依存性の注入

#### 優先度3: 信頼性

- エラーハンドリングとラッピング
- 並行処理（ゴルーチン、チャネル、Mutex）
- リソース管理（defer）
- テストの網羅性と品質

コードの変更に基づいて、関連するセクションを読み込んで体系的に評価する。

**重要**: レビュー中は常にこのファイルを読み込んで参照すること。内容を記憶に頼らない。

### 4. 変更ファイルごとの分析

diffの各ファイルについて：

1. **目的を理解する**: この変更は何を実現しようとしているのか？
2. **Goのイディオムと照合する**: "Effective Go" や "Go Code Review Comments" に従っているか？
3. **チェックリスト項目を適用する**: 関連するチェックリストのセクションを確認
4. **具体的な問題を記録する**: ファイルパス、行番号、問題の説明、重要度を記録

### 5. レビューレポートの生成

以下の「レポート形式」セクションで説明されている形式で包括的なレポートを作成する。

### 6. コード修正後の検証

指摘した問題をユーザーが修正した後、以下のコマンドを実行して、修正が正しく機能しているか検証する：

```bash
go build ./...
go test ./...
go fmt ./...
go vet ./...
```

エラーが発生した場合は、エラー内容を分析して追加の修正提案を行う。

## レポート形式

以下の構造でレビューレポートを生成する：

### サマリー

- **検出された問題の総数**: <数値>
- **Critical**: <件数>
- **High**: <件数>
- **Medium**: <件数>
- **Low**: <件数>
- **全体的な評価**: コード品質の簡潔な2〜3文のまとめ

### 重要度別の問題

各重要度レベル（Critical → High → Medium → Low）について、以下の形式で問題を列挙する：

#### [重要度] <問題カテゴリ>

**ファイル**: `<file_path>:<line_number>`

**問題**: 問題の明確な説明

**影響**: 潜在的な結果やリスクの説明

**現在のコード**:

```go
// 問題のあるコードスニペットを表示
```

**修正提案**:

```go
// 説明付きで修正されたコードを表示
```

**ベストプラクティス**: 追加のコンテキストや推奨されるパターン（該当する場合）

---

### 問題が見つからなかった場合

重要度カテゴリで問題が見つからなかった場合は、「✅ [重要度]の問題は見つかりませんでした」と記載する。

### 追加の所見

ポジティブな観察結果を含める：

- よく実装されているパターン
- 従われている良い慣行
- 称賛に値する改善

## リソース

### references/checklist.md

Goの基本的なアーキテクチャとコード品質に関する包括的なチェックリスト。このファイルには以下が含まれている：

- **可読性**: 命名規則、早期リターン、ドキュメント
- **拡張性・保守性**: インターフェース、パッケージ構成、Context、DI
- **信頼性**: エラーハンドリング、並行処理、リソース管理、テスト

各チェック項目は重要度（Critical/High/Medium/Low）で分類されており、レビューの優先度付けに役立つ。

**使用方法**: すべてのレビューの開始時にこのファイルを読み込み、コードの変更に基づいて関連するセクションを体系的に確認する。

## 効果的なレビューのためのヒント

1. **"Go Way" (イディオム) を尊重する**: シンプルさを追求し、不必要な抽象化を避ける
2. **エラーをファーストクラス市民として扱う**: エラーハンドリングはGoの核となる部分であり、無視や雑な処理は許容しない
3. **並行処理の落とし穴に注意する**: チャネルのクローズ忘れや、共有変数への同期なしのアクセスを厳格にチェックする
4. **パッケージの責務を明確にする**: 循環参照を避け、パッケージが単一の関心事を持つようにする
5. **具体例を提供する**: 問題を指摘するだけでなく、適切な修正方法（特にエラーラッピングやインターフェースの使い方）も示す
6. **GEMINI.md/CLAUDE.md を参照する**: プロジェクト固有のルールがある場合は、それを根拠に指摘を行う
7. **セキュリティ・パフォーマンスは別スキル**: セキュリティやパフォーマンスに関する問題を見つけた場合は、`security-performance-review`スキルの使用を推奨する
